## 질문 리뷰
- `static` 을 사용하던 것에 비해 동반객체를 활용하면서 어떤 이점 실무적으로 얻을 수 있을까요?
    - 개인적으로 도움이 된다고 생각함. 
    - `static` 같은 경우는 변수를 여러개를 만들면, 각자 독립적인 느낌 => 근데 동반 객체를 활용하면 여러 개를 묶어서 관리할 수 있는 느낌
    - 이것은 직접 Spring 등으로 애플리케이션을 개발하다보면 느낄 수 있음
- 팩토리 메서드에 대한 예제가 동반 객체 부분에 나왔던 것 같은데, 최상위 함수로 빼는 패턴과 동반객체로 활용하는 패턴 중 어느 것이 일반적으로 많이 사용되나요?
    - 케이스마다 다르다.
    - 전역 함수 => 코드가 깔끔함, 다른 파일에 있는 경우 어디에 있는 지 확인해야 하고, 캡슐화를 무너뜨리는 느낌
        - 코드를 이쁘게 쓰거나 클래스에 속해있는 것을 보여주고 싶을 때 의도적
        - 유틸성 클래스를 만들 때? 자주 활용함
    - 동반 객체 => 데이터에 속해있다는 것을 눈에 볼 수 있음
    - 휴리스틱에 가깝긴 함.

> 다른 분들의 질문 
- 확장 함수와 확장 변수는 JS의 동적 타이핑과 동일한 것 인가?
    - 자바의 경우 언어가 너무 딱딱해서 간단한 걸 하기 위해서도 여러 준비 코드가 많음. 그래서 코틀린이 지향하는 것은 자바를 좀 더 간단하게 쓰자. 이게 가장 큰 목표.
    - 그래서 해당 기능은 자바를 좀 더 편하게 쓰기 위한거인데, 그걸 위해서 다른 언어에서 필요한 기능을 가져온 경우가 있음. 그래서 JS 동적 타이핑이랑 비슷하다고 볼 수 있음.
- 클래스와 함수가 `final` 키워드가 기본적인 이유
    - impative Kotlin, impactive Java => 최대한 가변성을 최소화하라. `final`을 잘 쓰고, 상속을 제한적으로 써라 => 이런 철학 (가변성을 제약)
    - 그래서  `open`을 써서 해결하거나 해라(필요한 경우에만)
- `sealed class`와 `enum`
    - `sealed`는 보통 클래스에서 자식 클래스를 부모 클래스로 분기하면서 확인할 때 굳이 필요없는 `else`로 생기는 문제를 해결하기 위해서 추가된 기능. 
    - `enum`과 사용처가 다른 개념. 특히 이후 배울 내용에서 클래스만이 할 수 있는 기능이 있음. 
- 실무에서 코틀린의 자유분방함이 단점으로 오는 케이스가 있는가?
    - 예외처리 => 코틀린의 모든 예외가 언체크드 예외라. 이걸 놓칠 수 있음. (자바에서는 체크드 예외면 모두 처리해야하기 때문) => 개발자가 좀 더 신경써야함
    - 확장 함수/ 확장 변수 / 전역 함수 / 전역 변수 (특히 전역함수/변수) => 보통 단점으로 오는데, "어쩔 수 없이" 필요하다면 써라. => 이것 또한 컨벤션/규칙을 잘 정해서 쓰면 좋음
- 클래스의 함수를 언제 어디서나 선언할 수 있는데, 따로 제한하지 않는가?
    - 언어에서는 하지 않지만, 잘못쓰면 문제가 크기 때문에 컨변션 영역으로 가져가는 느낌
    - 특히 전역 함수/변수를 잘 처리해야 한다.
- 전체적으로 자바의 메커니즘이 어려워서 바꾼거다 => 라는 느낌인데, 전체적으로 이해가 잘 되지 않아 설명을 듣고 싶다.
- `internal`이 컴파일 될 때 `public`으로 된다고 했는데, 자바랑 코틀린이랑 같이 사용될 때 오히려 문제가 생기지는 않는가?
    - 생기긴 하지만, 흔한 케이스는 아님.
    - 보통 그런 것들은 주석을 잘 작성하거나 잘 관리함
    - 실제로는 자바와 코틀린을 같이 사용하는 경우는 잘 없고, 레거시 코드를 코틀린으로 변경하는 경우가 많음.
- 코틀린이 기본이 `final`인게 실제로 유지보수랑 도움이 되는 것인지?
    - 신규 개발에는 불편함. 
    - 최초에 상속을 고려하지 않은 클래스를 `open`으로 바꿔야하는 경우에 원천 객체를 수정해야 하는 경우가 많음
    - But, 불필요한 가변성, 즉 최초 개발할 때 상속 가능성을 고려하지 않고 설계했는데 => 추후 나중에 개발자가 확인해서 => 상속 안받게 코드를 작성할 수 있음 (코드만으로 의도 제안 가능(주석을 작성하지 않아도))
    - 즉, 설계적인 관점에서는 유리함
- 확장 함수랑 확장 프로퍼티 -> 둘 다 문법 상 멤버로 보여서 편한 것 같은데, 어디까지가 좋은 추상화인지, 캡슐화랑 코드 추적성을 헤치는 것인지?
