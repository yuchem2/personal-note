## 질문 리뷰
- `filter`에서 인덱스가 필요한 경우, Kotlin은 `filterIndexed`라는 별도 함수를 제공합니다. 반면 JavaScript/TypeScript는 같은 `filter` 콜백에서 두 번째 파라미터로 인덱스를 선택적으로 받을 수 있습니다. 또한, Python에서는 `enumerate`로 감싸서 해결합니다. Kotlin이 별도 함수로 분리한 설계 이유는 무엇이고, 어떤 방식이 더 낫다고 생각하시나요?
    - Kotlin은 Java와의 호환성을 위해 어쩔 수 없었음.. 파라미터의 오버로딩도 가능하지만 모호한 경우에 Kotlin에서 처리가 잘 되지 않음
    - Kotlin 입장에서는 `List<String>`과 `List<Int>`과 같은데, Java에서는 달라서 컴파일 하면 `JVM signature` 오류가 발생함 => 그래서 메서드명을 분리해서 여러 타입의 람다를 받을 수 있도록 만든 것.
---
- 지연 컬렉션 vs 즉시 컬렉션 어떤 것이 더 좋나요?
    - -> 답은 없고, 휴리스틱 하게 정의하게 된다. 
    - 컬렉션이 커지고 중간 과정이 커질 수록 성능 손해를 볼 수 있기 때문에 필요한 경우에 대부분의 경우에 `Sequence`를 활용하는 것이 더 좋다.
- 실무에서 함수 타입과 SAM을 나누는 경우?
    - -> 답은 없고, 휴리스틱하게 정의한다.
    - 상속하거나 분기하면서 사용할 것 같으면 SAM. 
    - 한번만 사용하거나 자주 사용하더라도 간단한거면 함수 타입으로 많이 함.
    - But, SAM을 쓰는 것이 항상 유리한 경우가 있음! -> 추후 설명
- `apply`, `also`, `with`, `run`, `let` 스코프 함수가 매우 유사한데, 어떤 상황에서 어떤 것을 쓰나요?
    - => 코틀린 스코프 함수는 5개가 있고, 보통 영어가 모국어인 사람이 느꼈을 때 상황에 맞는 것을 쓰면 적절함.
    - But, 우리는 자연스럽게 읽힌다를 느낄 수 없기 때문에.. 보통 컨벤션으로 정의함..
    - 스터디장님 팀에서는 `apply`, `also`를 많이 쓰고 `nullable`과 관련있으면 `let`

- 람다에서 오류를 잡을 때 `try-catch`? 다른 방법?
    - 람다를 중첩해서 쓸 때 일반적으로는 `try-catch`
    - 프레임워크라던가, 콜백이라던가, 이벤트리스닝이라던가 등의 경우에서는 함수 내에서 처리하거나 다른 방법들이 존재(`Either, Result 타입(공식 기능 x)`): [코틀린을 다루는 기술](https://product.kyobobook.co.kr/detail/S000001834373)
- 컬렉션 함수 중에는 DB 쿼리를 대체할만한 성능이 있는 함수가 있는가?
    - 없다고 보는 게 맞음. But DB 쿼리로 해야할 때 엄청 복잡해지는 경우나 단순한 `select`가 아니라 `if` 등이 있으면? 나을 수도 있음. 하지만 가능한 경우 DB 쿼리로 해결하는 게 맞음
- Kotlin에서 람다를 이용할 때 외부 변수를 `final`이 아니더라도 접근이 가능하도록 설계한 이유?
    - 정확한 답은 모르지만, 함수형 프로그래밍을 지원하기 위해서라고 들었음
    - 함수형에서는 일부로 사이드 이펙트를 일으켜야하는 경우도 있어서라고 들음
    - 언어 차원에서는 지원하지 않지만, 사이드 이펙트를 막고 싶으면 의도적으로 막을 수 있는 방법이 있
- Java의 `Stream`과 코틀린의 `Sequence`를 보면서 코틀린에서는 병렬 처리 어떻게 하나요?
    - 코틀린에는 코루틴이라는 좋은 기능이 있음. 그래서 코루틴이 병렬 기능을 잘 처리할 수 있기 때문에 병렬 처리를 위해 별도의 코드를 가지지 않고, 단순하게 지연 연산에만 신경썼음.

## 과제
> 마감: 2026.04.28 22:00
### 요구사항
1. **순수 코틀린 JVM 프로젝트로 작성**
2. **코틀린 버전은 2.2.20이상**
3. **Ktlint를 설치하고 엄격하게 코드 포멧팅을 지키기**
4. AI도움 받지 않고 순수 손으로 개발 및 궁금한 사항도 인터넷 검색을 쓸것
5. 로또는 **하나에 1000원**
6. 로또 번호는 **한 장에 6개**
7. 로또 숫자는 **1~45내 숫자 중복없이**
8. 각 회차별로 당첨 번호는 랜덤으로 정해야하며 각 회차의 당첨번호는 그 회차의 모든 로또 **구매 완료 이후 결정되어야함**
    1. 당첨번호는 6개 + 보너스 번호 1개로 이루어짐
9. 돈을 입금해서 해당 **한도 내에서 로또는 무제한 구매 가능**
    1. 예사) 10만원이 있을때 항상 10만원 어치를 구매하진 않을수도 있음 100장 이내로 몇개든 나눠서 구매 가능
10. **당첨금은 다시 입금내역**으로 들어감
    1. **1등 21억, 2등 6천만, 3등 150만, 4등 5만, 3등 5천**
    2. *1등 당첨번호 6개 일치, 2등 5개 일치 + 보너스 번호 일치, 3등 5개 일치, 4등 4개일치, 5등 3개 일치*
11. 로또 한장을 뽑는 방법은 **수동, 자동을 지원하며 수동, 자동 동시 구매 가능**
    1. 예시) 유저가 한번에 총 20장을 구매할때 *3장은 수동으로 입력, 17장은 자동으로 구매 할수 있어야함*
12. *회차는 시간으로 진행되어도 좋고 수동으로 개발자가 1회차씩 올리는것도 OK*
13. 각 회차의 결과는 저장되어야 하며 **프로그램을 껐다 켜면 이력을 자동으로 불러와서 다음 회차를 진행해야함**
14. **과거 회차의 결과를 조회할수 있는 기능**이 있어야함
15. Validation을 신경을 잘써야 하며 에러 처리도 잘해야함
16. **매직넘버의 사용은 지양하며 되도록 상수로 정의해야함**
17. **일급 컬렉션을 안쓰고 일단 작성해보고, 그걸 일급컬렉션 쓰는 코드로 리팩토링** 해볼것
18. 한개의 메소드가 **20줄이 넘지 않도록** 작성해야 함
19. 로또의 당첨확률이 매우 낮다는 사실을 직접 느껴보고 로또의 덧없음을 느낄것

### 선택사항
1. 로또 한장 번호 뽑는 방식에 반자동도 추가해볼것 (6개의 번호중에서 N개는 수동 6-N개는 자동)
2. 로또 한장 번호를 뽑는 방식을 전략패턴으로 각 전략을 갈아낄수 있는 방식으로 작성해볼것
3. 리드미도 잘 작성해볼것
4. 각 클래스는 명확히 구분되는 1개의 책임만을 가져야 하며 한개의 메소드는 한개의 역할만 수행하도록 작업
