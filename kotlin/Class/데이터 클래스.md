```kotlin
data class Customer(val name: String, val postalCode: Int)
```

어떤 클래스가 데이터를 저장하는 역할만을 수행한다면 [모든 클래스가 정의해야하는 메서드](모든%20클래스가%20정의해야하는%20메서드.md)를 반드시 오버라이드해야 한다. 코틀린은 `data` 변경자를 클래스 앞에 붙이면 필요한 메서드를 컴파일러가 자동으로 생성해 준다. 이러한 클래스를 **데이터 클래스**라고 한다.

컴파일러는 `equals`와 `hashCode`를 만들 때 주 생성자에 나열된 모든 프로퍼티를 고려한다. 생성된 `equals` 메서드는 모든 프로퍼티 값의 동등성을 확인한다. `hashCode` 메서드는 모든 프로퍼티의 해시 값을 바탕으로 계산한 해시 값을 반환한다. **이때 주 생성자 밖에서 정의된 프로퍼티는 `euqals`나 `hashCode`를 계산할 때 고려의 대상이 아니다.**

> **자바 레코드와 데이터 클래스**
> 자바 14에 레코드가 처음 도입되었다. 개념적으로 레코드는 여러 불변값으로 이뤄진 그룹을 다룬다는 점에서 코틀린 데이터 클래스와 유사하다. 레코드 또한 [모든 클래스가 정의해야하는 메서드](모든%20클래스가%20정의해야하는%20메서드.md)를 포함한 메서드를 자동으로 생성해 준다. 다만 `copy`와 같은 편의 메서드는 없다.
> 
> 코틀린과 비교해 더 많은 구조적 제약이 있다.
> - 모든 프로퍼티가 `private`이며 `final`이여야 한다.
> - 레코드는 상위 클래스를 확장할 수 없다.
> - 클래스 본문 안에서 다른 프로퍼티를 정의할 수 없다.
>   
>   상호운용성을 위해서 코틀린 data class에 `@JvmRecord` 어노테이션을 추가해 레코드를 선언할 수 있다. 이런 경우 데이터 클래스도 레코드 제약 사항을 따라야 한다.
## 불변성

데이터 클래스의 프로퍼티가 꼭 `val`일 필요는 없다. 하지만 데이터 클래스의 모든 프로퍼티를 읽기 전용으로 만들어 데이터 클래스를 불변 클래스로 만들라고 권장한다.

`HashMap`과 같은 컨테이너에 데이터 클래스 객체를 담는 경우에는 불변성이 필수적이다. 데이터 클래스 객체를 키로 하는 값을 컨테이너에 담은 다음 데이터를 바꾸게 되면 컨테이너 상태가 잘못될 수 있기 때문이다.

이를 위해 컴파일러는 `copy`라는 메서드를 제공한다. `copy`는 객체를 복사하면서 일부 프로퍼티를 바꿀 수 있게 해주는 메서드이다.
