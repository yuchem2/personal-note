기본적으로 코틀린 클래스에 대해 하위 클래스를 만들 수 없고, 기반 클래스의 메서드를 하위 클래스가 오버라이드할 수 없다. **즉 모든 클래스와 메서드는 기본적으로 `final`이다.** 이 방식은 *취약한 기반 클래스*라는 문제를 방지하기 위함이다.

> **취약한 기반 클래스**
> 기반 클래스 구현을 변경함으로써 하위 클래스가 잘못된 동작을 하게 되는 경우

만약 코틀린에서 어떤 클래스의 상속을 허용할려면 `open` 변경자를 앞에 붙여야 한다. 오버라이드를 허용하고 싶은 메서드나 프로퍼티의 앞에도 `open` 변경자를 붙여야 한다.

```kotlin
open class RichButton : Clickable {
    fun disable() { /* ... */ }
    open fun animate() { /* ... */ }
    override fun click() { /* ... */ }
}
```

만약 기반 클래스나 [인터페이스](인터페이스.md)의 멤버를 오버라이드한 경우에는 기본적으로 `open`으로 간주된다. 만약 이를 방지하기 위해서는 명시적으로 `final`을 붙여야 한다.

> **열린 클래스와 [스마트 캐스트](스마트%20캐스트.md)**
> 기본 상속 가능 상태를 `final`로 함으로써 얻을 수 있는 가장 큰 이점은 **스마트 캐스트**가 가능하다는 점이다. 프로퍼티가 `final`이 아니라면 그 프로퍼티를 다른 클래스가 상속하면서 커스텀 접근자를 저의함으로써 스마트 캐스트의 요구사항을 깰 수 있다.

클래스를 `abstract`로 선언할 수도 있다. 이 경우에는 추상 클래스라고 불리며 인스턴스화할 수 없다. 추상 클래스의 특징 상 추성 멤버는 항상 열려있다. (`open`, *이는 [인터페이스](인터페이스.md)의 멤버 앞에 `open`을 명시할 필요가 없는 것과 마찬가지이다.*)

```kotlin
abstract class Animated {
    abstract val animationSpeed: Double // 추상 프로퍼티
    val keyframes: Int = 20 // 추상이 아닌 프로퍼티는 final
    open val frames: Int = 60 // open으로 지정할 수 있음
    
    abstract fun animate()
    open fun stopAnimating() { /* ... */ }
    fun animateTwice() { /* ... */ }
}
```

| 변경자        | 이 변경자가 붙은 멤버는...             | 설명                            |
| ---------- | ---------------------------- | ----------------------------- |
| `final`    | 오버라이드할 수 없음                  | 클래스 멤버의 기본 변경자                |
| `open`     | 오버라이드할 수 있음                  | 반드시 `open`을 명시해야 오버라이드할 수 있다. |
| `abstract` | 반드시 오버라이드해야 함                | 추상 클래스의 멤버에만 이 변경자를 붙일 수 있다.  |
| `override` | 상위 클래스나 인터페이스의 멤버를 오버라이드하는 중 | 오버라이드하는 멤버는 기본적으로 열려있다.       |
