> 변하는 것은 캡슐화하고, 변하지 않는 것과 분리하라.

전략 패턴은 행동(알고리즘)을 별도의 클래스로 분리하여, 런타임에 동적으로 교체할 수 있게 해주는 행동 디자인 패턴
## 구조
```text
Context (맥락)
  └── Strategy (인터페이스)
        ├── ConcreteStrategyA
        ├── ConcreteStrategyB
        └── ConcreteStrategyC
```
- Context: 전략을 사용하는 객체. 전략을 직접 실행하지 않고 전략 객체에 위임
- Strategy: 모든 전략이 구현해야 할 공통 인터페이스
- ConcreateStrategy: 실제 알고리즘을 구현한 클래스
## 예시
```java
// 1. 전략 인터페이스
interface PaymentStrategy {
    void pay(int amount);
}

// 2. 구체적인 전략들
class CardPayment implements PaymentStrategy {
    public void pay(int amount) {
        System.out.println("카드로 " + amount + "원 결제");
    }
}

class KakaoPay implements PaymentStrategy {
    public void pay(int amount) {
        System.out.println("카카오페이로 " + amount + "원 결제");
    }
}

// 3. Context
class ShoppingCart {
    private PaymentStrategy strategy;

    public void setStrategy(PaymentStrategy strategy) {
        this.strategy = strategy;  // 런타임에 전략 교체 가능!
    }

    public void checkout(int amount) {
        strategy.pay(amount);
    }
}

// 4. 사용
ShoppingCart cart = new ShoppingCart();
cart.setStrategy(new CardPayment());
cart.checkout(10000);  // 카드로 10000원 결제

cart.setStrategy(new KakaoPay());
cart.checkout(10000);  // 카카오페이로 10000원 결제
```
## 장점 vs 단점

| 장점                              | 단점                  |
| ------------------------------- | ------------------- |
| OCP 준수 (새 전략 추가 시 기존 코드 수정 불필요) | 전략 클래스 수가 많아질 수 있음  |
| 알고리즘 교체가 유연                     | 클라이언트가 전략 차이를 알아야 함 |
| 조건문(if/switch) 제거 가능            | 단순한 경우엔 오버엔지니어링     |
| 테스트 용이                          |                     |
## 언제 사용하나요?
- 비슷한 동작을 하지만 다른 알고리즘이 여러 개 필요할 때
- 런타임에 동적으로 행동을 교체해야 할 때
- `if-else` /`switch` 분기가 지나치게 많을 때
- 정렬, 압축, 인증, 결제 등 교체 가능한 로직이 필요할 때
## 실제 사용 사례
- Java `Collections.sort(list, comparator)` -> Comparator가 전략
- Spring Security의 인증 전략
- 게임 캐릭터의 공격 방식 변경
- 파일 압축 방식 선택 (ZIP, GZIP, RAR 등)
