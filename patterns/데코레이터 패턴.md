객체에 추가적인 책임(기능)을 동적으로 부여하기 위한 구조 패턴. 상속의 대안으로 합성을 사용해서 기능을 확장한다.
## 왜 필요한가?

상속만으로 기능을 확장하려고 하면 다음과 같은 문제가 발생
1. 조합 과다: 커피 가게 메뉴를 모델링한다고 했을 때 에스프레소, 라떼, 아메리카노가 있고 여기에 휘핑크림, 시럽, 두유 옵션을 붙일 수 있으면 상속으로는 `EspressoWithWhip`, `EspressoWithWhipAndSyrup`, `LatteWithSoyAndSyrup` 같은 클래스를 모든 조합에 대해 옵션을 만들어야 함.
2. 정적 결정: 상속은 컴파일 타임에 결정되기 때문에 런타임에 동적 변경이 불가능

이 문제를 객체를 다른 객체로 감싸는 방법으로 해결하는 것이 데코레이터 패턴
## 구조

- Component: 원본 객체와 데코레이터가 모두 구현하는 공통 인터페이스
- ConcreteComponent: 실제 핵심 기능을 가진 원본 클래스
- Decorator: Component를 구현하면서 동시에 Component 인스턴스를 내부 필드로 가지는 추상 클래스. 모든 호출을 내부 객체에 그대로 위임하는게 기본 동작
- ConcreteDecorator: Decorator를 상속받아서 위임 전후로 추가 동작을 끼워 넣는 실제 데코레이터

## 예시

```java
// Component
interface Beverage {
    String getDescription();
    double getCost();
}

// ConcreteComponent
class Espresso implements Beverage {
    public String getDescription() { return "에스프레소"; }
    public double getCost() { return 3000; }
}

// Decorator
abstract class BeverageDecorator implements Beverage {
    protected Beverage beverage;
    public BeverageDecorator(Beverage beverage) {
        this.beverage = beverage;
    }
}

// ConcreteDecorator
class Whip extends BeverageDecorator {
    public Whip(Beverage beverage) { super(beverage); }
    public String getDescription() { return beverage.getDescription() + " + 휘핑"; }
    public double getCost() { return beverage.getCost() + 500; }
}

class Syrup extends BeverageDecorator {
    public Syrup(Beverage beverage) { super(beverage); }
    public String getDescription() { return beverage.getDescription() + " + 시럽"; }
    public double getCost() { return beverage.getCost() + 300; }
}

// 사용
Beverage drink = new Syrup(new Whip(new Espresso()));
// "에스프레소 + 휘핑 + 시럽", 3800원
```
## 실제로 사용되는 곳
- **Java I/O 스트림**
    - `new BufferedReader(new InputStreamReader(new FileInputStream("a.txt")))`
    - `FileInputStream`이 원본이고, `InputStreamReader`가 바이트→문자 변환을 추가하고, `BufferedReader`가 버퍼링을 추가
- GUI 라이브러리에서 컴포넌트에 스크롤바나 테두리를 입히는 경우
- Spring의 `BeanPostProcessor`
- AOP 프록시
- Python의 함수 데코레이터(`@decorator` 문법은 같은 아이디어를 함수 레벨에 적용한 것)
- HTTP 미들웨어 체인 등이 있어요.

## 장단점
- 장점
    - 런타임에 기능을 자유롭게 조합 가능
    - 단일 책임 원칙(SRP) 준수
    - 클라이언트 코드는 Component 인터페이스만 알면 되기 때문에 데코레이터의 존재를 몰라도 됨
- 단점
    - 데코레이터를 여러 겹 쌓으면 디버깅할 때 스택 트레이스가 복잡
    - 특정 데코레이터를 중간에서 빼거나 순서를 바꾸기 어려움
    - 데코레이터 적용 순서가 결과에 영항을 줄 수 있음
    - 데코레이터가 너무 많아지면 작은 객체들이 많이 생겨 시스템이 오히려 이해하기 어려워질 수 있음

## 비슷한 패턴과 차이
- 프록시: 구조는 거의 같지만 의도가 다름. 프록시는 접근 제어/캐싱/원격 호출 같은 "대리" 역할이 목적이고, 데코레이터는 "기능 추가"가 목적
- 어댑터: 인터페이스를 바꾸는 게 목적이지만, 데코레이터는 인터페이스를 그대로 유지
- 컴포지트: 트리 구조를 만드는 게 목적이지만, 데코레이터는 단일 객체를 한 겹씩 감싸는게 목적
