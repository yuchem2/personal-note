> 9장 학습 후 느낀점 공유 & 질문

- 9장의 중심은 "관례(convention)" — 약속된 이름의 함수를 정의하면 언어 구문(+, [], by 등)이 자동으로 그 함수 호출로 연결되는 메커니즘. 연산자 오버로딩과 위임 프로퍼티 모두 이 한 가지 메커니즘의 응용
- 연산자 오버로딩
    - 다른 언어들의 연산자 오버로딩을 코틀린식으로 통일한 느낌. 
        - 자바(전면 금지) ↔ C++(전면 허용) 사이의 절충안 같음. 정해진 연산자만, 정해진 이름으로만 허용.
- 위임 프로퍼티
    - 처음엔 "위임 클래스의 프로퍼티 버전인가" 싶었지만, 실제로는 둘 다 같은 관례 메커니즘 위에 있어서 한 장에 묶인 것으로 보임.
    - 연산자 오버로딩보다 실무에서 더 자주 쓰일 것 같다는 인상
## 질문 리뷰
- `plus`와 `plusAssign`을 둘 중 하나만 사용하라고 나와있는데, 둘 다 사용하는 경우가 있을 것 같다. 어떤 기준으로 사용할 수 있을까?
    - 스터디장님 부서는 두 개 다 씀. 하나만 쓰라는 건 좀..별로이지 않을까? 
    - 스터디장님 부서에서는 실제 돈을 많이 다루기 때문에 따로 돈을 클래스로 만들어서 연산하고 있어 실제로 `plus`와 `plusAssign`을 모두 정의해서 사용하고 있음.
- 위임 프로퍼티가 직접 `getter`와 `setter`를 지정하는 것보다 좋은 사례가 뭐가 있을까?
- 연산자 오버로딩을 언제 쓰는지? 함수가 더 낫지 않나?
- 코틀린은 왜 표준 숫자 타입에 비트 연산자를 정의하지 않고 함수만 정의했나요? 개인적으로는 기호가 있는 게 편하다고 느낌.
    - 스터디장님도 잘 모르겠다라고 하심..아마 자바에서도 없으니까 안 쓴 것 아닐까?
    - C++ 처럼 메모리 단위로 조절하고 그런 행위가 적어서 그런게 아닐까?
- 구조 분해 할당이 왜 순서로 설계되었는가?
    - JVM 기반으로 하다보니 그러지 않았을까?
    - 코틀린 컴파일 타임에 객체 순서를 비교하고, 이름을 같으면 다르게 하고? 그런게 복잡하지 않았을까?
    - 정적 언어일만큼 다이나믹하는게 안되지 않을까?
- 위임 프로퍼티는 위임 클래스보다 더 많이 쓰일 것 같고, 특히 프레임워크 차원에서 많이 제공할 것 같은데, 실제로 그런가? 자주 쓰는 위임 프로퍼티 사례가 있는지?
    - 뭐가 더 많이 사용된다는 것은 없다. 컨벤션이나 관성적으로 선택해서 사용하는 것이 많음. 설문을 하면 아마 5:5 정도지 않을까?
    - 실무 사례 (과금 정책): 
        - Set1~Set4, 전자문서 등 과금 이벤트 수와 처리 방식이 사소하게 다른 여러 케이스가 존재.
        - Set1: 무료 (이벤트 0) 
        - Set2: 한시적 무료 (이벤트 1, 0원 처리) 
        - Set3: 건당 N원 (이벤트 1) 
        - Set4: 이벤트별 다른 단가, VAT 포함 (이벤트 4) 
        - 전자문서: 이벤트별 다른 단가, VAT 별도 (이벤트 6)
        - 비슷한 구조에 사소한 차이가 있는 정책들이라 단순 구현 시 코드 중복이 늘어나고, 요금 변경 같은 사소한 수정도 여러 곳을 건드려야 함. → 위임을 활용해 공통 로직과 정책별 차이를 분리하는 방식으로 해결.
